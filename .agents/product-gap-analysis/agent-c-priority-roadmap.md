# 기능 우선순위 로드맵

## 사용자 목표
"맥미니에서 Claude/Codex 작업현황을 직원 사무실처럼 보고, 작업 완료 후 다음 지시를 이어가는 운영 시스템"

---

## Tier 1 - 즉시 필요 (1주 내)

### 1. 작업 완료 감지 (Task Completion Detection)
**우선순위**: 최상위
**근거**:
- 핵심 사용 시나리오의 출발점
- 이것 없이는 "작업 완료 후 다음 지시" 워크플로우가 불가능
- 다른 모든 기능(알림, 검수 상태, 다음 지시)의 선행 조건
- 구현 난이도: 중간 (로그 파싱 또는 API 폴링)

**구현 방향**:
- Claude Code/Codex의 상태 변화를 감지 (busy → idle)
- 로그 파일 모니터링 또는 프로세스 상태 체크
- 실시간 이벤트로 UI에 반영

**의존성**: 없음 (독립 기능)

---

### 2. 검수 필요 상태 표시 (Review Required Status)
**우선순위**: 최상위
**근거**:
- 작업 완료 감지 직후 필수적인 다음 단계
- 사용자가 "어떤 에이전트가 대기 중인지" 한눈에 파악 가능
- 일상 운영에서 가장 자주 보는 화면
- 구현 난이도: 낮음 (UI 상태 업데이트)

**구현 방향**:
- 에이전트 카드에 "Review Required" 배지 표시
- 마지막 작업 결과 미리보기
- 타임스탬프 표시 (얼마나 대기 중인지)

**의존성**: 작업 완료 감지

---

### 3. 알림 (Notifications)
**우선순위**: 높음
**근거**:
- 맥미니가 백그라운드에서 돌아갈 때 핵심
- 사용자가 다른 작업 중일 때도 작업 완료를 놓치지 않음
- 구현 난이도: 낮음~중간 (단계별 접근)

**구현 방향 (단계별)**:
- **1차**: 앱 내부 토스트 알림 (즉시 구현 가능)
- **2차**: 브라우저 Web Notifications API (권한 요청 필요)
- **3차**: PWA/iPad 알림 검토
- **후속**: macOS native notification 또는 Electron 도입 검토

**현실적 1차 선택**: 현재 Express + React 웹앱 구조에서는 브라우저 Web Notifications API가 가장 현실적

**의존성**: 작업 완료 감지

---

## Tier 2 - 단기 (2-4주)

### 4. 다음 지시 프롬프트 생성/복사 (Next Instruction Prompt)
**우선순위**: 높음
**근거**:
- 검수 후 다음 작업으로 자연스럽게 이어지는 워크플로우
- 반복 작업 시 생산성 크게 향상
- 구현 난이도: 중간 (컨텍스트 파싱 + 템플릿 생성)

**구현 방향**:
- 검수 완료 후 "다음 할 일" 제안
- 클립보드에 프롬프트 복사 버튼
- 템플릿 기반 프롬프트 생성 (프로젝트별 커스터마이징)
- 연속 작업 체인 지원 (A → B → C 시퀀스)

**의존성**: 검수 필요 상태 표시

---

### 5. 맥미니 상시 구동 (Always-on Service)
**우선순위**: 높음
**근거**:
- 실제 운영 환경에서 필수
- 재부팅 후에도 자동 시작
- 구현 난이도: 낮음 (launchd plist 설정)

**구현 방향**:
- launchd plist 파일 생성
- 로그인 시 자동 시작
- 크래시 시 자동 재시작
- 헬스 체크 및 자동 복구

**의존성**: 기본 시스템 안정성 확보 후

---

### 6. 프로젝트별 작업 큐 (Project-based Task Queue)
**우선순위**: 중간
**근거**:
- 여러 프로젝트 병렬 관리 시 유용
- 복잡한 워크플로우를 구조화
- 구현 난이도: 중간 (데이터 모델 확장)

**구현 방향**:
- 프로젝트별 작업 대기열
- 우선순위 설정 (긴급/일반/낮음)
- 드래그 앤 드롭으로 작업 순서 조정
- 작업 상태: 대기/진행중/완료/보류

**의존성**: 작업 완료 감지, 검수 상태 표시

---

## Tier 3 - 중기 (1-2개월)

### 7. iPad 외부 접속 (Remote Access)
**우선순위**: 중간
**근거**:
- 이동 중에도 에이전트 상태 확인 및 지시 가능
- 실제 "사무실처럼" 운영하려면 필요
- 구현 난이도: 중간 (네트워크 설정 + 보안)

**구현 방향**:
- Tailscale VPN으로 안전한 원격 접속
- 웹 인터페이스를 iPad에서 접속
- 반응형 UI (모바일 최적화)
- 선택적: Progressive Web App (PWA)

**의존성**: 기본 기능들이 안정화된 후

---

### 8. 에이전트 역할 고정 (Agent Role Assignment)
**우선순위**: 중간
**근거**:
- 전문화된 워크플로우 구축 시 유용
- 에이전트별 특성화로 효율 향상
- 구현 난이도: 중간 (설정 시스템 + UI)

**구현 방향**:
- 에이전트별 역할 태그 (코드 작성/검수/문서화/테스트)
- 역할에 맞는 프롬프트 템플릿
- 작업 자동 라우팅 (특정 역할에게만 할당)
- 역할별 성과 메트릭

**의존성**: 프로젝트별 작업 큐

---

## Tier 4 - 장기/선택

### 9. GitHub 이슈/PR 연동 (GitHub Integration)
**우선순위**: 낮음
**근거**:
- 고도로 자동화된 워크플로우에서 유용
- 현재 목표(기본 운영 시스템)에는 필수가 아님
- 구현 난이도: 높음 (GitHub API + 양방향 동기화)

**구현 방향**:
- GitHub 이슈에서 자동으로 작업 생성
- PR 리뷰 요청을 검수 작업으로 변환
- 작업 완료 시 이슈/PR 자동 업데이트
- Commit/PR 링크를 작업 히스토리에 기록

**의존성**: 프로젝트별 작업 큐, 에이전트 역할 고정

## 핵심 통찰 (Key Insights)

### 1. 초기 운영 완성 단계는 Tier 1 기능만으로 충분
"작업 완료 감지 → 검수 필요 상태 표시 → 알림"만 있어도 기본 워크플로우가 완성됩니다:
- 에이전트가 작업 완료
- 알림을 받고 확인
- 검수 후 다음 지시를 직접 입력

### 2. Tier 2는 생산성 배가
- 다음 지시 자동 생성: 반복 작업 시간 단축
- 상시 구동: 언제든 사용 가능
- 작업 큐: 복잡한 프로젝트 관리

### 3. Tier 3+는 스케일 업 단계
- 원격 접속: 장소 제약 제거
- 역할 고정: 다중 에이전트 협업
- GitHub 연동: 완전 자동화
