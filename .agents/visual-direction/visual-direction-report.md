# PixelOffice 시각 방향 보고서

**작성일:** 2026-07-02
**상태:** Codex 검수 통과 / 대안 A (Operations Map) 구현 완료

---

## 개요

현재 `PixelOffice`의 픽셀 캐릭터/귀여운 사무실/캐릭터 애니메이션 느낌을 유지할지, 아니면 더 운영 도구다운 관제형 시각화로 바꿀지 판단하기 위한 설계안입니다.

**참고:** 이 문서는 초기 분석 문서였으며, 후속 슬라이스에서 대안 A(Operations Map) 구현이 반영되었습니다.

## 현재 PixelOffice 분석

### 파일 구조

| 파일 | 라인 수 | 역할 |
|------|---------|------|
| `PixelOffice.tsx` | 735 | Canvas 기반 렌더링, 상호작용 |
| `PixelOffice.css` | 336 | 툴팁, 범례, 반응형 스타일 |

### 핵심 기능

#### 1. Agent 상태 표시
- **색상 코드**: idle(회색), working(녹색), waiting(주황)
- **투명도**: stale agent는 `0.4` 투명도
- **글로우 효과**: `needsReview` agent에 노란색 글로우
- **선택 표시**: 선택된 agent에 시안색 원형 하이라이트

#### 2. 프로젝트 분리
- `ProjectRoom` 인터페이스로 프로젝트별 방 구성
- 벽, 창문, 문 등 사무실 요소 렌더링
- 프로젝트명 라벨 표시

#### 3. 애니메이션
- `requestAnimationFrame` 기반 60fps 렌더링
- Agent 바운스 애니메이션 (working 상태)
- 부드러운 호버 전환

#### 4. 상호작용
- **마우스**: 호버 툴팁, 클릭 선택
- **터치**: 탭 선택
- **키보드**: Tab 이동, Enter/Space 선택

#### 5. 반응형
- Canvas 크기 동적 조정
- 모바일에서도 동작 (터치 지원)

### 장점

| 항목 | 설명 |
|------|------|
| 시각적 차별화 | 일반 관제 도구와 확연히 다른 UX |
| 친근함 | 픽셀 아트 스타일로 부담 감소 |
| 공간 활용 | 프로젝트별 방 배치로 자연스러운 그룹핑 |
| 애니메이션 | 상태 변화를 시각적으로 즉시 인지 |

### 단점

| 항목 | 설명 |
|------|------|
| 확장성 | Agent 50개 이상 시 레이아웃 복잡 |
| 정보 밀도 | 캐릭터당 표시 가능 정보 제한적 |
| 코드 복잡도 | Canvas 직접 렌더링으로 유지보수 부담 |
| 접근성 | 시각 장애인 접근성 제한 |

## 대안 A: Operations Map (운영 지도)

### 컨셉

지하철 노선도 스타일의 운영 현황 지도. 프로젝트를 역(station)으로, agent를 열차로 표현.

### 시각적 특징

```
[프로젝트 A] ────●────●────●──── [완료]
                ↑    ↑    ↑
              agent agent agent
              idle  work  wait

[프로젝트 B] ────●────●──── [완료]
```

- 수평 타임라인/진행선
- 원형 노드로 agent 표시
- 색상으로 상태 구분
- 점선/실선으로 연결 상태 표시

### 장점

- 진행 상태 직관적 파악
- 정보 밀도 높음
- 확장성 좋음 (스크롤로 대응)
- 접근성 향상 (DOM 기반 가능)

### 단점

- 친근함 감소
- 현재 픽셀 아트 스타일과 완전히 다름
- 구현 시 전면 재작성 필요

### 구현 난이도

| 항목 | 평가 |
|------|------|
| 코드 재사용 | 10% (로직만 재사용) |
| 예상 작업량 | 대 (전면 재작성) |
| CSS/HTML | DOM 기반으로 전환 |

## 대안 B: Terminal Grid (터미널 그리드)

### 컨셉

터미널/콘솔 스타일의 텍스트 기반 그리드. 모노스페이스 폰트와 ASCII 아트 활용.

### 시각적 특징

```
┌─────────────────────────────────────────┐
│ PROJECT: agent-control-center           │
├─────────────────────────────────────────┤
│ [●] feature-auth   WORKING  2m  Claude  │
│ [○] fix-bug-123    IDLE     5m  Codex   │
│ [!] refactor-ui    REVIEW   1h  Claude  │
└─────────────────────────────────────────┘
```

- 고정폭 폰트
- 박스 드로잉 문자
- 컬러 코드 (ANSI 스타일)
- 스캔라인 효과 (선택적)

### 장점

- 정보 밀도 최고
- 개발자 친화적
- 텍스트 기반으로 접근성 우수
- DOM 기반으로 유지보수 쉬움

### 단점

- 시각적 매력 감소
- 비개발자에게 불친절
- 현재 스타일과 완전히 다름

### 구현 난이도

| 항목 | 평가 |
|------|------|
| 코드 재사용 | 20% (상태 로직, 이벤트 핸들러) |
| 예상 작업량 | 중 (DOM 기반 새 컴포넌트) |
| CSS/HTML | 새 스타일시트 필요 |

## 대안 C: Mission Control (미션 컨트롤)

### 컨셉

NASA 미션 컨트롤 스타일. 현재 픽셀 아트의 친근함을 유지하면서 정보 밀도 향상.

### 시각적 특징

```
┌────────────────────────────────────────────────┐
│  ╔═══════════════════════════════════════════╗ │
│  ║  PROJECT: agent-control-center            ║ │
│  ╠═══════════════════════════════════════════╣ │
│  ║  [픽셀캐릭터1] [픽셀캐릭터2] [픽셀캐릭터3] ║ │
│  ║   feature      fix-bug      refactor     ║ │
│  ║   WORKING      IDLE         REVIEW       ║ │
│  ╚═══════════════════════════════════════════╝ │
│                                                │
│  ╔═══════════════════════════════════════════╗ │
│  ║  PROJECT: another-project                 ║ │
│  ...                                           │
└────────────────────────────────────────────────┘
```

- 픽셀 캐릭터 유지 (축소/간소화)
- 프로젝트별 패널 카드
- 상태 정보 텍스트 병행 표시
- 그리드 레이아웃

### 장점

- 픽셀 아트 친근함 일부 유지
- 정보 밀도 향상
- 확장성 개선 (카드 그리드)
- 점진적 마이그레이션 가능

### 단점

- 두 스타일의 혼합으로 일관성 저하 가능
- 픽셀 캐릭터 축소 시 가독성 저하
- 구현 복잡도 증가 (Canvas + DOM 혼합)

### 구현 난이도

| 항목 | 평가 |
|------|------|
| 코드 재사용 | 60% (캐릭터 렌더링, 상태 로직) |
| 예상 작업량 | 중 (레이아웃 재구성) |
| CSS/HTML | 기존 CSS 확장 |

## 권장 사항

### 단기 (유지)

현재 PixelOffice를 유지하되, 다음 개선 고려:

1. **Agent 제한**: 화면당 최대 30개 표시, 나머지는 축소 뷰
2. **툴팁 강화**: 더 많은 정보 표시
3. **키보드 탐색 개선**: 프로젝트 간 이동 단축키

### 중기 (점진적 전환)

Mission Control(대안 C) 방향으로 점진적 전환:

1. 프로젝트별 카드 레이아웃 도입
2. 픽셀 캐릭터 크기 축소하여 정보 텍스트 병행
3. Canvas → DOM 부분 전환

### 장기 (선택적)

Agent 수가 급증하거나 모바일 중심이 되면:

- Terminal Grid(대안 B) 또는 Operations Map(대안 A) 전면 전환 검토
- 사용자 설정으로 뷰 모드 선택 기능

## 결론

현재 PixelOffice는 시각적 차별화와 친근함에서 강점이 있습니다. 그러나 agent 수 증가 시 확장성과 정보 밀도에서 한계가 있습니다.

**최종 선택:** Operations Map(대안 A) 구현. DOM 기반 그리드 레이아웃으로 접근성, 정보 밀도, 유지보수성을 개선했습니다.

**구현 완료 항목:**
- `client/src/components/OperationsMap.tsx` - 신규 컴포넌트
- `client/src/components/OperationsMap.css` - 신규 스타일
- `client/src/App.tsx` - PixelOffice → OperationsMap 교체

**최종 상태:** Codex 검수 통과 / 구현 완료

## 부록: 현재 PixelOffice 핵심 코드 구조

### 렌더링 파이프라인

```
requestAnimationFrame
  └─ draw()
       ├─ renderFloor()
       ├─ renderProjectRooms()
       │    ├─ renderWalls()
       │    ├─ renderWindows()
       │    └─ renderProjectLabel()
       ├─ renderAgents()
       │    ├─ renderAgent() (각 agent)
       │    │    ├─ 상태별 색상
       │    │    ├─ needsReview 글로우
       │    │    ├─ 선택 하이라이트
       │    │    └─ 바운스 애니메이션
       │    └─ hitTest() (상호작용용)
       └─ renderUI()
            └─ 툴팁, 범례
```

### 주요 상태

```typescript
interface Agent {
  id: string
  name: string
  status: 'idle' | 'working' | 'waiting'
  needsReview?: boolean
  isStale?: boolean
  projectPath?: string
  source?: 'claude' | 'codex'
}
```

### 프로젝트 방 계산

```typescript
interface ProjectRoom {
  projectPath: string
  x: number
  y: number
  width: number
  height: number
  agents: Agent[]
}
```
